**Версия:** 1.0 · Июнь 2026

---

## Принципы ролевой модели

**Каждый участник — пользователь продукта.** Обязательные условия с первого дня:

- Установить бот, заполнить профиль, подписаться на канал БЦ
- Минимум один обед в неделю через платформу — покрывается еженедельным БЛ-сертификатом
- Ежедневно вести дневник питания (модуль 4)
- Раз в месяц — аудит-визит (когда запустится модуль 6)
- Регулярно слать фидбек «от пользователя» в командный канал

## Роли управления

### 1. Project Manager

**Ветка:** Project Management

Отвечает за процесс: сроки, синхронизацию, ритуалы, блокеры. Удерживает команду в темпе и не даёт модулям разъехаться. Не управляет продуктом — это зона Product Manager.

**Задачи:**

- Ведение бэклога в трекере (Linear / Notion / kaiten)
- Спринт-планирование, дейли, ретро — регулярность и структура
- Контроль дорожной карты пилота (месяцы 1–6 из операционного пакета)
- Управление зависимостями между модулями (например: Mini App не начинается без готового API)
- Ведение таблицы рисков (часть X операционного пакета) и эскалация блокеров
- Onboarding новых участников: бриф, доступы, первый дейли
- Коммуникация со Школой 21 и Клубом технологического предпринимательства

**Как использует продукт:** отслеживает retention на себе — если сам перестал открывать бот, это сигнал проблемы с продуктом.

---

### 2. Product Manager

**Ветка:** Product Management

Отвечает за продукт: что строим, зачем, для кого и как меряем успех. Работает в паре с Project Manager: один управляет «что», другой — «как и когда».

**Задачи:**

- User research: интервью с сотрудниками СанСити, CustDev до и после запуска
- Формирование user stories и acceptance criteria для каждого модуля
- Приоритизация бэклога совместно с Project Manager
- Определение метрик успеха для каждого модуля (OKR на пилот)
- Мониторинг Retention D30, MAU, воронки конверсии канал → бот → дневник
- Юзабилити-тесты совместно с UX-дизайнером (тестирование на реальных пользователях)
- A/B тесты гипотез совместно с аналитиком
- Roadmap на следующие фазы (согласованно с Project Manager)

**Как использует продукт:** анализирует собственный пользовательский опыт как первичный источник инсайтов перед каждым спринтом.

## Аналитика

### 3. Бизнес-системный аналитик

**Ветка:** Бизнес и системный анализ

Отвечает за то, чтобы все понимали систему одинаково: бизнес — бизнес-процессы, разработчики — спецификации.

**Задачи:**

- Технические спецификации для каждого модуля (функциональные и нефункциональные требования)
- ER-диаграммы: сущности системы (заведение, меню, блюдо, пользователь, КБЖУ-запись, аудит)
- Моделирование бизнес-процессов: сбор меню → верификация → дайджест → уведомление
- Юнит-экономика: LTV, CAC, ARPU, breakeven по одному БЦ
- Требования к интеграциям: Telegram Bot API, LLM Vision, база КБЖУ
- Документация API совместно с backend-разработчиком
- Когортный анализ и воронки удержания совместно с Product Manager
- Описание монетизационных сценариев для переговоров с заведениями

**Как использует продукт:** анализирует собственные логи и поведение как пользователя; делает первые A/B тесты на аудитории команды.

## Разработка

> Три роли разработки работают параллельно на разных слоях и интегрируются через общий API-контракт. Контракт фиксируется аналитиком на старте каждого спринта.

### 4. Backend-разработчик

**Ветка:** Разработка

Ядро системы: данные, логика, вычисления. Не трогает Telegram-специфику и парсеры — это отдельные роли.

**Задачи:**

- Проектирование схемы БД: заведения, меню, пользователи, КБЖУ-дневник, профили предпочтений
- REST API для Mini App и внутренних сервисов (OpenAPI-спецификация)
- Бизнес-логика формирования персонального дайджеста (фильтрация по профилю: ограничения, бюджет, антипредпочтения)
- Алгоритм рекомендаций: ранжирование топ-3 заведений для пользователя
- Расчёт КБЖУ: сложение записей дневника, вычисление дефицита/профицита относительно нормы
- Планировщик задач: утренняя рассылка в 11:00 (дайджест канала), пуш пользователям в 11:30, вечерняя сводка КБЖУ
- Логика фоллбэка: если нет данных на сегодня — выводить последнее известное меню с пометкой о дате
- Эндпоинты для бота администратора заведения (обновить меню, подтвердить данные)
- Очереди для массовых рассылок (Redis + Celery / RQ)

**Как использует продукт:** до релиза каждой фичи — обязательный прогон в роли обычного пользователя через бота.

---

### 5. Разработчик Telegram Bot + Mini App

**Ветка:** Разработка

Клиентский слой: всё, что видит и трогает пользователь. Интегрируется с Backend через API.

**Задачи — Telegram Bot:**

- Конечный автомат (FSM): состояния пользователя, переходы между сценариями
- Обработка команд: `/start`, `/lunch`, `/ate`, `/water`, `/today`, `/settings`
- Онбординг-сценарий: пошаговый ввод профиля (вкусы, ограничения, бюджет, привязка к БЦ)
- Inline-клавиатуры и кнопки быстрого ответа (быстрая запись «Пообедал здесь»)
- Отправка персональных push-уведомлений в 11:30 (получение данных из Backend, форматирование карточки)
- Обработка ошибок и тупиков FSM (пользователь ушёл на полпути)

**Задачи — Telegram Mini App (WebApp):**

- Экран «Сегодня»: список доступных ланчей с фильтрами (цена, кухня, расстояние)
- Экран дневника питания: добавление записей, просмотр дня, прогресс по КБЖУ
- Экран истории: где обедал, средний чек, выполнение нормы
- Экран профиля: редактирование ограничений, антипредпочтений, цели по КБЖУ
- Интеграция Mini App с Backend API (авторизация через Telegram initData)
- Адаптация под мобильный Telegram-клиент (iOS / Android)

**Как использует продукт:** ежедневно пользуется и ботом, и Mini App; находит UX-проблемы раньше пользователей.

---

### 6. Инженер данных (парсинг + LLM)

**Ветка:** Разработка

Невидимая инфраструктура: откуда берутся данные о меню. Без этого весь продукт пустой.

**Задачи:**

- Интеграция LLM Vision (Claude Vision / GPT-4o Vision / local modeling): распознавание состава меню и цены с фотографий, формирование дата-сета и верификация разметки
- ETL-пайплайн: collect (парсинг) → extract (LLM) → normalize (единый формат блюда) → verify (запрос заведению) → store (в БД через Backend API)
- Telegram-бот для администраторов заведений: форма «опубликовать меню на сегодня» (альтернативный push-канал)
- Верификация: после парсинга + кнопка подтверждения где-то как-то
- Интеграция с базой КБЖУ (FoodData Central / FatSecret / собственная): сопоставление блюд из меню с составом нутриентов
- Мониторинг полноты: ежедневный отчёт — по каким заведениям данных нет → триггер фоллбэка в Backend

**Как использует продукт:** проверяет точность распознавания на собственных обедах — сравнивает автоматически распознанный состав с реальной тарелкой.

## Качество и безопасность

### 7. QA-инженер

**Ветка:** Тестирование

Гарантирует, что ежедневный дайджест в 11:00 выходит точно и без ошибок. Это репутационная ставка.

**Задачи:**

- Тест-план для каждого модуля (согласованно с Product Manager и аналитиком)
- Ручное тестирование бота: все команды, переходы FSM, граничные сценарии (пустой профиль, нет ланчей рядом)
- Ручное тестирование Mini App: все экраны, фильтры, редактирование профиля
- Тест-кейсы для фоллбэков: нет данных о меню → корректная пометка; заведение не ответило → не пустая карточка
- Тестирование парсеров: замер точности LLM Vision vs ручная верификация (target: &gt;90% точности)
- Тестирование дайджеста: формат карточки заведения, корректность данных, время публикации (11:00 ±2 мин)
- Нагрузочное тестирование: одновременная рассылка 300+ пользователям — очередь не падает
- Автоматизированные API-тесты (Postman / pytest): эндпоинты Backend, граничные значения
- Регрессионное тестирование при каждом деплое
- Баг-репорты в бэклог (приоритет: критичное → дайджест не вышел; высокое → неверное меню; среднее → UX-проблемы)

**Как использует продукт:** ведёт отдельный тестовый аккаунт и личный аккаунт — сравнивает опыт для выявления environment-специфичных проблем.

---

### 8. Специалист по информационной безопасности

**Ветка:** Тестирование

Защищает персональные данные пользователей и репутацию платформы. Особенно критичен при запуске модуля 8, где обрабатываются данные о питании и здоровье сотрудников компаний.

**Задачи:**

- Аудит соответствия 152-ФЗ: классификация персданных (профиль питания — специальная категория), реестр обработки, политика конфиденциальности, согласия
- Анализ безопасности интеграции с Telegram API: хранение bot-токенов, webhook security (HTTPS + secret token), валидация initData в Mini App
- Penetration testing API: SQL-инъекции, auth bypass, rate limiting, IDOR (доступ к чужому профилю)
- Безопасность хранения данных в БД: шифрование, минимизация привилегий, логирование доступов
- Аудит LLM-интеграций: что именно отправляется в сторонний API (фото меню — без лиц, без геоданных пользователя)
- Контроль обезличивания HR-аналитики (модуль 8): валидация, что агрегированный отчёт не позволяет идентифицировать конкретного сотрудника
- Security review кода разработчиков (чек-лист для PR: токены, логирование, валидация входных данных)
- Подготовка документации для юридической рамки (совместно с аналитиком): обработка персданных аудиторов, поваров, HR-клиентов

**Как использует продукт:** регулярно проверяет приватность на собственных данных; документирует каждую уязвимость, найденную на своём аккаунте.

### 9. UX/UI-дизайнер

**Ветка:** Дизайн

Проектирует опыт: как пользователь взаимодействует с ботом и Mini App. Работает в паре с Product Manager (что проектировать) и Разработчиком Bot+App (как реализовать).

**Задачи:**

- Карта пользовательских путей (user flows): онбординг, получение дайджеста, запись в дневник, поиск по фильтрам
- Wireframes и интерактивные прототипы Mini App в Figma
- Дизайн-система для Telegram Mini App: цвета, типографика, иконки, компоненты
- Макеты карточек заведений для канала БЦ (шаблон публикации)
- Дизайн онбординг-сценария в боте (тексты кнопок, структура вопросов)
- Юзабилити-тесты: живые сессии с сотрудниками СанСити, фиксация проблем
- UX-аудит каждого экрана Mini App раз в спринт

**Как использует продукт:** проходит каждый пользовательский сценарий минимум раз в неделю, фиксирует боли в командном канале.

## Инфраструктура

### 10. DevOps-инженер

**Ветка:** Dev-ops

Ответственность номер один: дайджест в 11:00 не имеет права не выйти. Это единственный продуктовый KPI DevOps.

**Задачи:**

- Выбор и настройка хостинга (VPS / облако): изоляция prod / staging окружений
- Контейнеризация всех сервисов (Docker + Docker Compose / Kubernetes при росте)
- CI/CD пайплайн: автоматический деплой при мерже в main, прогон тестов QA как gate
- Настройка брокера очередей (Redis) и воркеров (Celery / RQ) для массовых рассылок
- Мониторинг: алёрты при сбоях сервисов, алёрт если дайджест не опубликован до 11:05
- Логирование и трассировка: централизованные логи всех сервисов (ELK / Loki)
- Резервное копирование БД: ежедневный бэкап, проверка восстановления раз в неделю
- Управление секретами: env-переменные, токены, ключи LLM (не в репозитории)
- HTTPS для всех эндпоинтов и webhook Telegram

**Как использует продукт:** замеряет время доставки push-уведомления на свой аккаунт каждое утро — это живой healthcheck системы.

## Роли без прямой ветки Школы 21

> Роли ниже не имеют прямого соответствия ни одной из веток. Рекомендуются студентам, которые совмещают техническое обучение с предпринимательским интересом, или студентам других образовательных программ вуза-партнёра.

### 11. Маркетолог

**Ближайшая ветка по духу:** Product Management  
**Старт:** Месяц 2 (после стабилизации канала БЦ и базового бота)

Отвечает за рост аудитории и медиа-присутствие проекта. На MVP маркетинг — это в первую очередь правильный старт в СанСити; глобальная реклама не нужна.

**Задачи:**

- GTM-план для запуска в СанСити: точки входа (чаты арендаторов, физические объявления в холле, внутренние каналы ДубльГИС и Школы 21)
- Ведение городского Telegram-канала: контент-план, 2–3 поста в неделю, редакционный голос
- Привлечение первых 300 подписчиков канала БЦ (рост органики + партнёрства)
- Рекламные интеграции для заведений в городском канале
- Аналитика контента: охваты, CTR, рост подписчиков
- Брендинг: голос «Обедыча», тон, визуальный стиль публикаций

**Как использует продукт:** тестирует каждый рекламный пост на себе до публикации; фиксирует расхождения между тем, что обещаем, и тем, что получает пользователь.

### 12. Менеджер по ключевым клиентам (заведения)

**Ближайшая ветка по духу:** Бизнес и системный анализ  
**Старт:** День 1

Единственный, кто физически обходит заведения. Без него нет базы — без базы нет дайджеста.

**Задачи:**

- Обход 15–20 заведений в радиусе 800 м от СанСити в первые 2 недели: фото меню, фиксация контактов, переговоры о подключении
- Обучение администраторов заведений работе с ботом для публикации меню
- Подписание договоров о размещении (совместно с юристом)
- Продажа платных тарифов Стандарт (10 БЛ/мес) и Премиум (20 БЛ/мес)
- Удержание платящих клиентов: регулярный контакт, обработка возражений
- Продажа пакетов аудита (модуль 6) и контент-продакшна (модуль 7) после запуска
- Эскалация операционных проблем с заведениями (не обновляют меню, не отвечают на верификацию)

**Как использует продукт:** обедает в заведениях, которые сам подключил; тестирует процесс обновления меню глазами администратора заведения.
